### 3. 수행 작업 내용 (Work Performed)
#### 코드 개발/구현
* 구현 1: 비동기 데이터 스트리밍 (Topic)
Turtlesim 및 Teleop 노드를 활용하여 geometry_msgs/msg/Twist 인터페이스 기반의 cmd_vel 발행.
ros2 bag을 활용하여 토픽 데이터를 기록하고 재생하여 경로 재현 테스트 수행.
성과: Fire-and-Forget 방식의 비동기 단방향 통신 흐름 확인.

* 구현 2: 제어 흐름의 동기화와 상태 관리 (Service vs Action)
Service: /spawn, /kill 서비스를 호출하여 거북이 객체를 제어. Request 후 Response를 대기하는 동기식(Blocking) 처리 확인.
Action: /rotate_absolute 액션을 통해 Goal 전송 후 Result를 수신(Service 방식)하고, 이동 중 Feedback으로 남은 각도(remaining)를 실시간 수신(Topic 방식)하는 복합 통신 구조 구현.

* 구현 3: 분산 시스템의 관측 가능성 확보 (Observability with RQt)
rqt_graph로 노드 간 연결 관계 시각화, rqt_plot으로 데이터 변화 추적, Service Caller 플러그인으로 파라미터 동적 업데이트 테스트.

### 4. 문제점 및 해결 과정 (Issues & Solutions)
* 문제점 1: Action 인터페이스 구조 파악 혼동
현상: turtlesim의 기능을 분석하던 중, 입력값으로 theta를 넣었는데 출력값 설명에 delta가 포함되어 있어 자료형 파악에 혼선 발생.
원인 및 해결: 확인 결과, turtlesim/srv/Spawn 서비스(x, y, theta, name)와 turtlesim/action/RotateAbsolute 액션(goal: theta, result: delta)의 인터페이스 정의를 혼동함.
조치: ros2 interface show 명령어를 통해 Service(srv)와 Action(action) 파일의 구조 차이를 명확히 구분하여 정리함.

* 문제점 2: N:N 통신 테스트 환경 제약
현상: 물리적인 PC 부족으로 다중 기기 간 N:N 통신 테스트에 어려움.
해결 방안: ROS2는 동일 네트워크 또는 로컬 환경에서도 Node 간 N:N 통신을 지원함. 추후 멀티 터미널을 활용해 다수의 Subscriber 노드를 실행하는 방식으로 N:N 통신 부하 테스트를 진행할 계획.



### 5. 배운점 및 다음 단계 (Learned & Next Steps)
* 배운점:
Topic은 비동기 연속 통신, Service는 동기식 요청-응답 통신임을 명확히 이해함.
Action은 Goal State Machine을 기반으로 Service(목표/결과)와 Topic(피드백)이 결합된 하이브리드 통신 방식임을 학습함.

* 다음 단계:
단일 PC 환경에서 다수의 노드를 띄워 Topic N:N 통신 시 데이터 지연(Latency)이나 손실이 발생하는지 부하 테스트 진행 계획.
